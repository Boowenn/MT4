# MT5 Pending-Order Worker Design

This is a design document only. The current branch must not add a Python
pending-order worker, order endpoint, close endpoint, cancel endpoint, or any
other trading path. The MT5 Dashboard APIs remain read-only until this design
is explicitly approved and implemented in a separate branch.

## Scope

Potential future scope:

- Convert approved research candidates into pending-order intents.
- Submit MT5 pending orders through one owned execution path.
- Track broker tickets, lifecycle state, fills, cancels, expirations, and
  realized outcomes.
- Write an immutable audit ledger before and after every external MT5 action.

Non-goals:

- No market orders in the worker.
- No `/api/mt5-readonly/order`, `/api/mt5-readonly/close`, or cancel route.
- No credential storage in the Dashboard.
- No mutation of live EA presets.
- No competing trading authority between EA and Python.

## Trading Authority

The highest-risk decision is ownership. Before implementation, choose exactly
one active owner for live MT5 order placement:

- `EA_ONLY`: current safer default. Python and Dashboard remain read-only.
- `PY_PENDING_ONLY`: Python may place pending orders, but EA must not place the
  same route/symbol pending orders.
- `EA_AND_PY_SPLIT`: only allowed if route/symbol ownership is locked in a
  signed config and both sides enforce it.

The first implementation should start with `EA_ONLY` plus dry-run ledger
replay. Moving to `PY_PENDING_ONLY` requires a separate release gate.

## Kill Switch

All order mutation must be blocked unless every switch is green:

- `QG_MT5_PENDING_WORKER_ENABLED=true`
- `QG_MT5_PENDING_WORKER_DRY_RUN=false`
- `QG_MT5_PENDING_AUTH_LOCK_PATH` exists and contains the expected lock token.
- Dashboard safety status says `orderSendAllowed=false` for read-only APIs and
  does not expose mutation endpoints.
- A local operator lock file confirms this host owns MT5 trading for the target
  account, server, route, and symbol.

Any one red switch forces dry-run/no-send mode. The worker should re-check the
kill switch before every external MT5 call, not only at startup.

## Authorization Lock

The authorization lock should be a local file outside the repo, generated by an
operator action, never by the Dashboard UI.

Suggested fields:

- `lockId`
- `createdAtIso`
- `expiresAtIso`
- `host`
- `accountLogin`
- `server`
- `mode`
- `allowedRoutes`
- `allowedCanonicalSymbols`
- `maxOrdersPerDay`
- `maxLotsPerOrder`
- `maxTotalLots`
- `operator`
- `reason`
- `signature`

The worker must refuse expired, mismatched, or unsigned locks.

## Audit Ledger

Every decision writes an append-only row before any MT5 mutation.

Suggested ledger: `QuantGod_MT5PendingOrderLedger.csv`

Columns:

- `LedgerId`
- `IntentId`
- `DecisionTimeIso`
- `Mode`
- `DryRun`
- `AccountLogin`
- `Server`
- `Route`
- `CanonicalSymbol`
- `BrokerSymbol`
- `OrderType`
- `Lots`
- `EntryPrice`
- `StopLoss`
- `TakeProfit`
- `ExpirationTimeIso`
- `Reason`
- `RiskSnapshotJson`
- `AuthLockId`
- `KillSwitchSnapshotJson`
- `Decision`
- `BrokerOrderTicket`
- `BrokerRetCode`
- `BrokerComment`
- `WorkerVersion`

The audit write must happen before `order_send`. After the broker response, the
same ledger row or a follow-up event row records the broker ticket/retcode.

## Dry Run

Dry run is the default and must support full lifecycle simulation:

- Generate order intents from the same candidate input as live mode.
- Validate symbol mapping, spread, margin, size, stop distance, and session.
- Write `Decision=DRY_RUN_ACCEPTED` or `DRY_RUN_REJECTED`.
- Never import or call the MT5 mutation function in dry-run-only tests.
- Compare dry-run outcomes against later market movement before promotion.

## Limits

Hard limits should be enforced at worker level even if upstream research says
the trade is valid:

- Max pending orders per account/day.
- Max pending orders per route/symbol/day.
- Max lots per order.
- Max total open plus pending lots per canonical symbol.
- Max total portfolio lots.
- Min distance from current bid/ask.
- Max spread.
- Max allowed slippage for conversion or fill handling.
- News/session blocks.
- Daily loss and drawdown blocks.

Limits should be included in both the authorization lock and worker config. The
stricter value wins.

## State Machine

Suggested states:

- `INTENT_CREATED`
- `DRY_RUN_ACCEPTED`
- `DRY_RUN_REJECTED`
- `AUTHORIZED`
- `ORDER_SEND_REQUESTED`
- `ORDER_SEND_ACCEPTED`
- `ORDER_SEND_REJECTED`
- `BROKER_ORDER_OPEN`
- `FILLED`
- `EXPIRED`
- `CANCEL_REQUESTED`
- `CANCEL_ACCEPTED`
- `CANCEL_REJECTED`
- `CLOSED_BY_EA_OR_BROKER`
- `AUDIT_MISMATCH`
- `KILL_SWITCH_BLOCKED`

Each transition needs an audit event and idempotency key.

## Idempotency

The worker must derive a stable `IntentId` from:

- account/server
- route
- canonical symbol
- broker symbol
- order side/type
- entry/SL/TP/expiration
- source candidate id

Before sending an order, it checks the ledger for an existing active intent or
broker ticket. Duplicate intents are skipped and audited.

## Implementation Gate

Before any implementation branch:

- Contract tests prove `/api/mt5-readonly/*` and
  `/api/mt5-symbol-registry/*` still cannot mutate MT5.
- Dry-run unit tests cover kill switch, auth lock, limits, idempotency, and
  audit writes.
- Manual smoke uses a demo account only.
- A rollback plan confirms disabling one env var stops all worker sends.
- The owner model is approved: `EA_ONLY`, `PY_PENDING_ONLY`, or
  `EA_AND_PY_SPLIT`.
